# 简历描述与面试讲解话术

## 1. 简历项目描述

项目名称：工业视觉检测结果复核与质量闭环管理系统

项目描述：

基于 Spring Boot 3 + Vue 3 设计并实现工业视觉质检结果复核与质量闭环管理系统原型，模拟 YOLO 检测结果进入企业质量流程后的人工复核、NCR 不合格记录、CAPA 整改闭环和批次状态追踪。系统覆盖生产批次、检测任务、检测图片、检测结果、人工复核、NCR、CAPA 等核心对象，支持从模型 JSON 导入到批次质量状态关闭的完整业务链路。

技术栈：

- 后端：Java 21、Spring Boot 3、MyBatis-Plus、MySQL 8、Knife4j、Validation、Lombok。
- 前端：Vue 3、Vite、Element Plus、Vue Router、Axios、ECharts。

个人职责：

- 设计 8 张核心业务表，明确批次、任务、图片、检测结果、复核、NCR、CAPA 的追溯关系。
- 实现 YOLO JSON 导入逻辑，将模型输出的类别、置信度、bbox 坐标和原始 payload 转换为业务数据。
- 实现人工复核模块，将模型检测结果与人工复核结论分离，支持确认缺陷、误检和需复检。
- 实现 NCR / CAPA 质量闭环模块，使用事务保证创建 NCR、创建 CAPA、关闭 CAPA 时跨表状态一致。
- 搭建 Vue 3 前端页面，完成 Dashboard、批次、任务、检测结果、检测结果详情、复核、NCR、CAPA 的主链路展示与操作。
- 实现 Dashboard 统计看板，展示批次、检测结果、待复核、NCR、CAPA 等质量指标。
- 实现检测结果详情页与 bbox 可视化，用样例图片叠加检测框展示类别、置信度和复核状态。
- 编写数据库设计、状态机、API 清单、演示脚本和面试说明文档。

项目亮点：

- 不是简单 CRUD，而是围绕工业视觉质检后的质量闭环建模。
- 引入 `PENDING_REVIEW -> CONFIRMED_DEFECT -> NCR -> CAPA -> CLOSED` 的可追溯流程。
- 将 AI 模型输出、人工复核结论和质量业务单据分层存储，便于扩展和审计。
- 关键流程使用事务保障 `detection_result`、`review_record`、`ncr_record`、`capa_record`、`production_batch` 状态一致。
- Phase 2 增加 Dashboard 质量看板和 bbox 可视化详情页，让系统既能展示质量闭环状态，也能直观展示工业视觉检测结果。

可放入简历的精简版：

> 工业视觉检测结果复核与质量闭环管理系统：基于 Spring Boot 3、MyBatis-Plus、MySQL 8、Vue 3 和 Element Plus 实现工业视觉质检业务原型，支持 YOLO JSON 检测结果导入、bbox 可视化详情、人工复核、NCR 不合格记录、CAPA 整改闭环、Dashboard 质量看板和批次状态追踪。设计 8 张核心业务表和状态机，使用事务保证 NCR/CAPA 创建与关闭时跨表状态一致，完成从模型检测结果到质量闭环关闭的完整主链路。

## 2. 面试讲解话术

### 2.1 一分钟版本

这个项目是一个工业视觉质检结果复核与质量闭环系统。它不是训练 YOLO 模型，而是解决模型检测结果进入企业质量管理后的业务流程问题。

主链路是：生产批次 -> 检测任务 -> YOLO JSON 导入 -> 检测结果 -> 人工复核 -> NCR -> CAPA -> 批次关闭。

我重点做了三件事：

1. 把 YOLO 输出的检测框、类别和置信度转换成业务表里的检测结果。
2. 把模型结果和人工复核结论分离，只有人工确认的缺陷才能进入 NCR。
3. 通过 NCR 和 CAPA 模块实现质量闭环，并用事务保证 CAPA 关闭时同步关闭 NCR 和批次状态。

### 2.2 三分钟版本

这个项目的背景是：视觉模型可以识别缺陷，但企业质量管理不能只依赖模型输出。真实场景里，检测框还需要人工复核，确认后才能形成不合格记录，再进入整改和验证流程。

所以我把系统拆成几层：

- 第一层是生产批次和检测任务，负责承载业务对象和模型执行记录。
- 第二层是检测图片和检测结果，负责保存 YOLO JSON 导入后的结构化结果。
- 第三层是人工复核，负责把模型结果转成人工结论。
- 第四层是 NCR 和 CAPA，负责质量问题记录、整改措施和关闭验证。

状态机上，检测结果先是 `PENDING_REVIEW`，人工复核后变成 `CONFIRMED_DEFECT`、`FALSE_POSITIVE` 或 `NEED_RECHECK`。只有 `CONFIRMED_DEFECT` 可以创建 NCR。创建 NCR 后批次进入 `NCR_OPEN`；创建 CAPA 后批次进入 `CAPA_OPEN`；关闭 CAPA 后，系统同步关闭 CAPA、NCR 和批次。

技术实现上，后端使用 Spring Boot 3、MyBatis-Plus 和 MySQL，前端使用 Vue 3、Element Plus 和 Axios。关键业务操作都放在 Service 层，Controller 只负责接收请求和返回统一结果。创建 NCR、创建 CAPA、关闭 CAPA 都使用事务，保证跨表状态一致。

### 2.3 可以重点展开的问题

问题：为什么不直接把 YOLO 结果当成缺陷？

回答：

> 因为模型输出只是候选结果，工业质量结论通常需要人工确认。系统把 `detection_result` 和 `review_record` 分开，既保留模型证据，也保留人工判断。这样误检不会直接进入 NCR，质量流程更可靠。

问题：NCR 和 CAPA 的关系是什么？

回答：

> NCR 是不合格记录，表示已经确认存在质量问题；CAPA 是纠正和预防措施，用来分析原因、制定措施并验证效果。系统中只有 OPEN 状态 NCR 可以创建 CAPA，同一个 NCR 只能创建一条 CAPA。

问题：项目里事务用在哪里？

回答：

> 事务主要用在跨表状态一致的地方。例如创建 NCR 时要插入 `ncr_record` 并更新批次状态为 `NCR_OPEN`；创建 CAPA 时要插入 `capa_record`，同时更新 NCR 为 `CAPA_CREATED`、批次为 `CAPA_OPEN`；关闭 CAPA 时要同步关闭 CAPA、NCR 和批次。这些动作不能只成功一部分。

问题：这个项目和普通 CRUD 的区别是什么？

回答：

> 普通 CRUD 只关注单表增删改查，而这个项目围绕一条工业质量业务链路建模：模型输出、人工确认、不合格记录、整改闭环和批次状态回写。核心价值在状态机、追溯链路和跨表一致性。

问题：如果继续迭代，会做什么？

回答：

> 我会优先补充登录权限、复核历史、多图片导入、NCR/CAPA 审批流、真实图片文件存储、模型推理服务集成、检测框多框联动展示，以及更完整的质量趋势分析。

## 3. Phase 2 面试话术

### 3.1 为什么加 Dashboard？

回答：

> Phase 1 已经完成了质量闭环的业务链路，但面试或演示时，如果只看列表页面，面试官需要自己拼接系统状态。Dashboard 的作用是把批次、检测结果、待复核、NCR、CAPA 等关键指标集中展示出来，让系统一打开就能看到当前质量流程处在哪些状态。它也体现了真实企业系统常见的“运营看板”能力。

可以补充：

- `batchCount`、`taskCount`、`detectionCount` 说明业务规模。
- `pendingReviewCount` 说明待人工处理工作量。
- `openNcrCount`、`inProgressCapaCount` 说明质量问题和整改压力。
- 状态分布图说明数据不是孤立列表，而是能被汇总分析。

### 3.2 为什么加 bbox 可视化？

回答：

> 检测结果如果只展示 bbox 坐标，其实更像普通后台表格。bbox 可视化让检测结果回到工业视觉场景本身：在哪里发现缺陷、是什么类别、置信度是多少、现在是否已经复核。即使用的是演示图片，也能让面试官直观看到“视觉检测结果进入质量业务系统”的连接点。

可以补充：

- bbox 坐标按固定逻辑尺寸映射到图片显示尺寸。
- 坐标缺失或越界时页面不会崩溃，体现了前端容错。
- 详情页保留人工复核入口，减少从视觉证据到业务处理的跳转成本。

### 3.3 这个项目和普通后台管理系统的区别是什么？

回答：

> 普通后台管理系统通常围绕单表 CRUD 或简单审批流展开，而这个项目围绕工业视觉质检后的质量闭环建模。它有模型输出、人工复核、NCR、CAPA 和批次状态回写，每个环节都有明确的状态约束和追溯关系。核心不是“能增删改查”，而是“模型检测结果如何变成可审计、可关闭的质量业务流程”。

可以强调：

- 不是直接把模型结果当最终结论，而是经过人工复核。
- 不是孤立记录缺陷，而是追溯到批次、任务和图片。
- 不是创建单据就结束，而是通过 CAPA 关闭回写批次状态。
- 关键业务使用事务保证跨表一致性。

### 3.4 如何体现工业视觉质检场景？

回答：

> 项目里保留了视觉模型输出的核心元素：类别、置信度、bbox 坐标和 raw payload；同时把这些结果接入质量管理对象：生产批次、检测任务、人工复核、NCR 和 CAPA。详情页的 bbox 可视化负责展示“视觉证据”，质量闭环负责展示“业务处置”，这两部分结合起来就比普通 QMS 或普通 YOLO demo 更贴近工业视觉质检场景。

可以举例：

- `inspection_task` 表示一次模型检测任务。
- `inspection_image` 表示被检测图片。
- `detection_result` 保存类别、置信度和 bbox。
- `review_record` 保存人工判断。
- `ncr_record` 和 `capa_record` 保存质量单据和整改闭环。

### 3.5 还有哪些可扩展方向？

回答：

> 后续可以沿三个方向扩展：第一是更真实的视觉数据接入，包括图片上传、文件存储、多图片任务和真实推理服务；第二是更完整的质量流程，包括审批、责任人、超期提醒、复核历史和 CAPA 验证记录；第三是数据分析能力，包括缺陷趋势、批次质量对比、模型误检率、复核效率和 CAPA 关闭周期。

可扩展清单：

- 接入对象存储或本地文件服务，展示真实检测图片。
- 支持一个图片多个 bbox 的批量可视化。
- 增加模型版本维度，分析不同模型的误检率和缺陷命中率。
- 增加复核历史与审计日志。
- 增加 NCR / CAPA 审批流和超期预警。
- 增加 Dashboard 趋势图，例如每日缺陷数、OPEN NCR aging、CAPA 平均关闭周期。
- 接入权限体系，区分质检员、质量工程师、CAPA 负责人和管理员。

## 4. 演示顺序建议

1. 打开 Dashboard，说明质量统计看板和当前闭环状态。
2. 打开批次列表，说明批次是质量追溯入口。
3. 从批次详情查看检测任务。
4. 从检测任务跳转检测结果。
5. 点击检测结果详情，展示 bbox 可视化。
6. 对 `PENDING_REVIEW` 检测结果提交人工复核。
7. 在复核记录中创建 NCR。
8. 在 NCR 页面创建 CAPA。
9. 在 CAPA 页面编辑、待验证并关闭 CAPA。
10. 回到批次和 NCR 页面确认状态同步关闭。

## 5. 项目边界说明

- 本项目不是完整 MES / QMS。
- 不包含真实企业数据。
- 不做在线模型推理。
- 不接入真实工业相机或产线设备。
- 当前重点是后端主链路、前端演示页面和质量闭环建模。
