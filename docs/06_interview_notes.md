# 简历描述与面试讲解话术

## 1. 简历项目名称

中文：工业视觉检测结果复核与质量闭环管理系统

英文：Visual Quality Flow System

## 2. 一句话项目描述

基于 Spring Boot 3 + Vue 3 实现的工业视觉质检全栈项目原型，用于模拟 YOLO 检测结果进入质量管理流程后的人工复核、NCR 不合格记录、CAPA 整改闭环、Dashboard 质量看板和 bbox 可视化展示。

## 3. 简历项目描述

项目描述：

面向工业视觉质检场景，设计并实现一个质量闭环管理系统原型，用于模拟 YOLO 检测结果进入企业质量管理流程后的人工复核、不合格记录、整改闭环和批次质量状态追踪。系统覆盖生产批次、检测任务、检测图片、检测结果、人工复核、NCR、CAPA 等核心对象，支持从模型检测结果导入到质量闭环关闭的完整演示链路。

技术栈：

Java 21、Spring Boot 3、MyBatis-Plus、MySQL、Vue 3、Element Plus、Axios、Vue Router、ECharts、Knife4j、Maven、Docker Compose、JUnit 5、MockMvc、Git。

主要工作：

- 设计核心业务表，建立生产批次、检测任务、检测图片、检测结果、人工复核、NCR、CAPA 之间的追溯关系。
- 实现 YOLO JSON 导入接口，将模型输出的类别、置信度、bbox 坐标和 raw payload 转换为业务检测记录。
- 实现人工复核流程，将模型检测结果与人工复核结论分离，支持确认缺陷、误检和需复检。
- 实现 NCR / CAPA 质量闭环，覆盖不合格记录创建、整改措施创建、验证关闭和批次状态回写。
- 在创建 NCR、创建 CAPA、关闭 CAPA 等关键流程中使用事务，保证跨表状态一致。
- 实现 Dashboard 质量看板，统计批次、检测任务、检测结果、待复核、NCR、CAPA 等质量指标。
- 实现检测结果 bbox 可视化详情页，用样例图片叠加检测框展示类别、置信度、状态和复核入口。
- 补充 Docker Compose MySQL 初始化、核心流程 MockMvc 集成测试、README、演示脚本和项目截图，提升项目可运行性和展示完整度。

可放入简历的精简版：

> 工业视觉检测结果复核与质量闭环管理系统：基于 Spring Boot 3、MyBatis-Plus、MySQL、Vue 3 和 Element Plus 实现工业视觉质检业务原型，支持 YOLO JSON 检测结果导入、bbox 可视化详情、人工复核、NCR 不合格记录、CAPA 整改闭环、Dashboard 质量看板和批次状态追踪。设计核心业务表和状态流转，使用事务保证 NCR/CAPA 创建与关闭时跨表状态一致，并通过 Docker Compose 和 MockMvc 集成测试提升项目可运行性。

## 4. 1 分钟面试讲解

这个项目是一个工业视觉质检场景的全栈原型，后端用 Spring Boot 3 和 MyBatis-Plus，前端用 Vue 3 和 Element Plus。

它不是训练 YOLO 模型，而是模拟 YOLO 检测结果进入质量管理系统之后怎么流转。主流程是：生产批次、检测任务、YOLO JSON 导入、检测结果、人工复核、NCR、CAPA，最后回写批次质量状态。

我重点做了三块：第一，把 YOLO 输出的类别、置信度和 bbox 转成业务检测记录；第二，把模型检测结果和人工复核结论分开，避免误检直接进入质量流程；第三，用 NCR 和 CAPA 做质量闭环，并用事务保证 CAPA 关闭时同步关闭 NCR 和批次。Phase 2 又补了 Dashboard 质量看板、bbox 可视化详情页、Docker Compose 和核心流程测试，所以这个项目更适合作品集和面试演示。

## 5. 3 分钟面试讲解

这个项目的背景是工业视觉检测。YOLO 这类模型可以输出缺陷类别、置信度和检测框坐标，但在真实质量管理语境里，模型输出通常只是候选结果，不能直接等同于最终质量结论。所以我做的不是模型训练，而是模型结果进入质量管理系统后的业务闭环。

系统主链路是：生产批次 -> 检测任务 -> YOLO JSON 导入 -> 检测结果 -> 人工复核 -> NCR 不合格记录 -> CAPA 整改闭环 -> 批次关闭。批次是追溯入口，检测任务代表一次模型检测执行，检测结果保存类别、置信度、bbox 和 raw payload，人工复核负责把模型候选结果转成人工结论，NCR 和 CAPA 负责后续质量问题记录和整改关闭。

它和普通 CRUD 的区别在于，核心价值不是单表增删改查，而是状态流转和追溯关系。比如检测结果先是 `PENDING_REVIEW`，人工确认后才可能变成 `CONFIRMED_DEFECT`；只有确认缺陷才能创建 NCR；只有 OPEN 状态的 NCR 能创建 CAPA；关闭 CAPA 时需要同步关闭 CAPA、NCR 和生产批次。创建 NCR、创建 CAPA、关闭 CAPA 这些跨表操作都放在事务里，避免只成功一部分导致质量状态不一致。

模型检测结果和人工复核结果分开，是因为模型可能误检。`detection_result` 保留模型证据，`review_record` 保留人工判断，这样既方便追溯，也避免误检直接污染 NCR/CAPA 流程。

Phase 2 我补了两个展示能力：Dashboard 和 bbox 可视化。Dashboard 用于展示批次、检测结果、待复核、OPEN NCR、进行中 CAPA 等指标，让系统打开后能先看到质量闭环的状态。bbox 可视化详情页把表格里的坐标叠加到样例图片上，让面试官能直观看到缺陷位置、类别、置信度和复核入口，更贴近工业视觉质检场景。

工程化方面，我补了 Docker Compose MySQL 初始化，方便一键准备演示数据库；也补了 MockMvc 集成测试，用 HTTP 接口串起 Dashboard、YOLO JSON 导入、人工复核、NCR、CAPA 创建与关闭，以及批次/NCR/CAPA 状态同步。这个项目定位是学习、作品集和面试展示用原型，不是完整 MES/QMS，也没有接入真实产线。

## 6. 高频面试问题与回答

### 这个项目解决什么问题？

它解决的是视觉模型检测结果进入质量管理流程后的业务流转问题。模型输出 bbox、类别和置信度后，还需要人工复核、质量问题记录、整改措施和批次状态追踪。本项目把这条链路做成一个可演示的全栈原型。

### 为什么不是普通 CRUD？

普通 CRUD 更关注单表数据维护，而这个项目关注质量闭环：模型结果、人工确认、NCR、CAPA 和批次状态回写。它有明确的状态约束、追溯关系和跨表一致性要求。

### 为什么需要人工复核？

模型检测结果可能有误检或不确定情况，不能直接作为质量结论。人工复核把候选检测结果转成业务结论，只有 `CONFIRMED_DEFECT` 才能进入 NCR，能减少误检对质量流程的影响。

### 为什么要设计 NCR 和 CAPA？

NCR 表示已经确认的不合格记录，CAPA 表示针对质量问题的纠正和预防措施。两者组合起来，能从“发现问题”走到“分析原因、制定措施、验证关闭”，更贴近质量管理流程。

### 关闭 CAPA 时为什么要同步关闭 NCR 和批次？

CAPA 关闭代表整改和验证完成。此时关联的 NCR 应该关闭，对应批次的质量状态也应该回到 CLOSED。否则会出现 CAPA 已关闭但 NCR 或批次仍显示未关闭的状态不一致。

### 如何保证跨表状态一致性？

关键流程使用 Spring 事务。例如创建 NCR 时插入 NCR 并更新批次状态；创建 CAPA 时插入 CAPA、更新 NCR、更新批次；关闭 CAPA 时同步关闭 CAPA、NCR 和批次。任一步失败都会回滚，避免部分成功。

### YOLO JSON 导入做了什么？

导入接口接收模拟 YOLO 输出，解析 sourceName、className、confidence、bboxXyxy 等字段，创建或复用检测图片记录，并把每个 box 写入检测结果表，同时保留 raw payload，方便后续追溯。

### Dashboard 统计了什么？

Dashboard 统计生产批次、检测任务、检测结果、待复核数量、确认缺陷数量、OPEN NCR、进行中 CAPA、已关闭 CAPA，并按批次状态、检测结果状态、缺陷类别、NCR 状态和 CAPA 状态做分布展示。

### bbox 可视化有什么价值？

它把表格里的 bbox 坐标变成图片上的可视化检测框，让缺陷位置、类别、置信度和复核状态更直观。对工业视觉质检项目来说，这比单纯展示坐标更容易说明业务场景。

### Docker Compose 在项目里起什么作用？

Docker Compose 用来启动 MySQL 8，并通过初始化 SQL 自动创建数据库表和演示数据。它降低了项目运行门槛，方便面试或换环境时快速准备数据库。

### 自动化测试覆盖了什么？

测试使用 JUnit 5、Spring Boot Test 和 MockMvc，覆盖 Dashboard 统计接口，以及创建批次、创建检测任务、导入 YOLO JSON、人工复核、创建 NCR、创建 CAPA、关闭 CAPA、校验批次/NCR/CAPA 状态同步的核心流程。

### 项目目前有哪些不足？

目前没有登录权限和复杂角色控制，没有真实产线接入，没有在线模型推理，图片展示使用样例图，也还没有完整审计日志、报表导出和部署流水线。这些是原型项目的边界。

### 后续可以怎么扩展？

可以扩展登录权限、审计日志、真实图片上传、对象存储、一个图片多个 bbox 的可视化、模型版本效果统计、报表导出、NCR/CAPA 审批流、超期提醒、CI 自动测试和部署脚本。

## 7. 项目不足与后续扩展

当前项目边界：

- 没有登录权限和复杂角色控制。
- 没有接入真实工业相机、产线设备或 MES 系统。
- 没有在线模型推理，YOLO 结果通过 JSON 导入模拟。
- 图片展示使用样例图，未实现真实图片上传和存储。
- 尚未实现完整操作日志、审计追踪、报表导出和部署上线流程。
- 自动化测试依赖已启动并初始化的 MySQL 环境，尚未引入 Testcontainers。

后续可扩展方向：

- 增加登录、权限和角色控制，区分质检员、质量工程师和管理员。
- 增加审计日志，记录复核、NCR、CAPA 状态变更。
- 接入真实图片上传和对象存储，支持多图片、多 bbox 可视化。
- 接入模型推理服务或异步任务队列，模拟更接近真实产线的数据流。
- 增加质量趋势报表、缺陷类别趋势、CAPA 关闭周期和 NCR aging 分析。
- 增加 CI 流程，在提交或 PR 时自动执行后端编译、测试和前端构建。

这个项目定位是学习、作品集和面试展示用原型，重点展示工业视觉检测结果进入质量流程后的建模能力、状态流转设计和全栈实现能力。
